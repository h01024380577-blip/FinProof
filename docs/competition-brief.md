## 1. 공모전 맥락

JB금융그룹 Fin:AI Challenge는 JB금융그룹의 사업 방향성과 연결되는 AI Agent 서비스를 기획하고 개발하는 대회다. 참가자는 지정주제 또는 자유주제 중 하나를 선택하며, 공통적으로 AI가 상황과 데이터를 이해하고, 필요한 정보를 바탕으로 판단하며, 적절한 행동이나 지원을 수행하는 Agent형 서비스를 지향한다.

전체 주제는 다음 네 가지다.

| 구분       | 분류               | 상세 주제                                      |
| ---------- | ------------------ | ---------------------------------------------- |
| 지정주제 1 | 개인 라이프케어 AI | LifeLong WM(Wealth Management) AI Agent 서비스 |
| 지정주제 2 | Compliance AI      | 준법자문가 AI Agent 서비스                     |
| 지정주제 3 | 마케팅 AI          | 디지털 마케팅 AI Agent 서비스                  |
| 자유주제   | JB 미래사업 AI     | JB금융그룹 연계 AI Agent 서비스                |

FinProof는 지정주제 2 `Compliance AI`에 해당한다. 대회 주제 문서에서 Compliance AI는 대고객 콘텐츠 준법심의가 수작업 중심이고, 다국어·다채널 확장 시 심의 리소스가 선형적으로 늘어나며, 규제 변경 반영이 늦어지는 문제를 해결하는 방향으로 제시되어 있다.

## 2. 제품 정체성

서비스명은 `FinProof Agent`다.

한 줄 정의:

> 금융 홍보물의 준법 리스크를 근거로 검증하고 수정안을 제안하는 AI Agent

조금 더 긴 정의:

> 요청자의 심의 의뢰부터 준법심의자의 최종 판단까지 하나의 플랫폼에서 연결하는 근거 기반 준법심의 워크플로우 솔루션

중요한 제품 원칙:

- AI는 최종 승인자가 아니다.
- AI는 위험 지점, 관련 근거, 수정·반려 의견 초안을 먼저 정리한다.
- 최종 조치인 `승인 / 수정요청 / 반려 / 보류`는 준법심의자가 결정한다.
- 모든 판단은 법령, 사내 기준, 상품자료, 과거 사례 등 추적 가능한 Evidence에 묶여야 한다.
- 결과뿐 아니라 자료 제출, AI 분석, 근거 확인, 최종 판단까지 AuditEvent로 남겨야 한다.

## 3. 해결하려는 문제

금융 홍보물 심의는 콘텐츠 적시성과 리스크 통제의 접점이다. 앱 배너, 푸시, 랜딩페이지, SNS, 제휴 채널, 다국어 안내 등 금융 콘텐츠 접점은 늘어나지만, 심의 과정은 여전히 자료 확인, 위험 표현 검토, 근거 탐색, 의견 작성이 수작업으로 이어지는 구조다.

핵심 병목은 `근거 확인`이다.

- 홍보물, 약관, 상품설명서, 체크리스트, 금리표, URL 등 자료가 흩어져 있다.
- 심의자는 표현 리스크와 상품 조건, 필수 고지, 내부 기준, 과거 사례를 계속 대조해야 한다.
- 심의자별 판단 편차와 휴먼에러가 발생할 수 있다.
- 심의 지연으로 마케팅 콘텐츠의 적시성이 떨어진다.
- 규제 변경이 있어도 현업 심의 기준에 즉시 반영되기 어렵다.
- 다국어 콘텐츠는 번역 뉘앙스와 현지 표현 리스크까지 추가로 검토해야 한다.

FinProof의 해법은 AI가 반복 검토와 근거 탐색을 선처리하고, 사람이 최종 판단에 집중하도록 하는 것이다.

## 4. 사용자와 권한

FinProof는 하나의 플랫폼에서 세 역할이 협업하는 구조다.

| 역할               | 사용자                        | 주요 행동                                               |
| ------------------ | ----------------------------- | ------------------------------------------------------- |
| `requester`        | 심의 요청자, 마케팅·제작 부서 | 심의 패키지 제출, 진행 상태 확인, 수정 요청 반영        |
| `reviewer`         | 준법심의자                    | AI 이슈 확인, 근거 검토, RAG 질의, 최종 판단, 의견 작성 |
| `compliance_admin` | 준법관리자, 기준·지식 관리자  | 지식문서 승인, 심의 기준 관리, 규제 변경 확인           |

권한은 RBAC로 분리해야 한다. 요청자는 자기 요청과 결과를 중심으로 보고, 심의자는 워크벤치에서 검토·판단하며, 관리자는 지식베이스와 심의 기준의 품질을 관리한다.

## 5. 핵심 업무 흐름

제품의 기본 흐름은 다음과 같다.

1. 요청자가 홍보물·상품자료·약관·체크리스트 등을 패키지로 업로드한다.
2. 시스템이 파일을 안전하게 해제하고, 홍보물·약관·상품설명서·체크리스트·URL 등으로 자동 분류한다.
3. 상품군별 필수 자료 누락 여부를 확인하고, 누락 시 보완 요청으로 제출을 막거나 심의 시작을 지연시킨다.
4. OCR로 문서와 이미지의 텍스트를 추출한다.
5. 지식문서와 과거 사례를 검색하고 리랭킹해 Evidence를 확정한다.
6. 도메인 Sub-Agent가 병렬 검토한다.
7. Main Compliance Agent가 검토 의견을 통합하고 충돌을 조정한다.
8. 고위험 또는 저신뢰 이슈는 보수적으로 재검증한다.
9. 심의자는 워크벤치에서 위험 이슈, 하이라이트, 근거 패널, RAG 채팅, 의견 초안을 검토한다.
10. 심의자가 최종 판단을 확정한다.
11. 산출물, 의견서, 근거 목록, 결정 이력, AuditEvent가 저장된다.

외국어 구간이 포함되면 AI 선분석과 근거 확인 사이에 다국어 리스크 매핑 단계가 추가된다.

## 6. AI 분석 파이프라인

기능명세서와 제안서의 AI 분석 구조는 3단계다.

### STEP 1. 근거 준비

- OCR 텍스트 추출
- 쿼리 임베딩
- 지식문서 검색
- 과거 심의 케이스 검색
- 리랭킹
- Evidence 확정

이 단계의 원칙은 LLM이 먼저 결론을 내리지 않게 하는 것이다. 먼저 검색된 근거와 업로드 자료를 기준으로 검토 범위를 좁힌다.

### STEP 2. Agent 동시 검토

Sub-Agent는 도메인별로 분리한다.

| Sub-Agent           | 검토 관점                                                        |
| ------------------- | ---------------------------------------------------------------- |
| `creative_review`   | 홍보물 문구, 문맥, 과장·오인 가능성, 표현 적합성                 |
| `product_terms`     | 상품 조건, 약관 위반, 금리·수수료·우대조건·고지사항 누락         |
| `regulation`        | 금융광고 관련 법령과 규정, 금지·주의 표현                        |
| `internal_policy`   | 사내 심의 가이드라인, 체크리스트 충족 여부                       |
| `case_search`       | 과거 유사 승인·반려·수정 사례                                    |
| `multilingual_risk` | 영어·베트남어·미얀마어·크메르어 구간의 번역 뉘앙스와 오인 리스크 |

Sub-Agent 의견이 상충하거나 근거 신뢰도가 낮으면 Main Compliance Agent가 재검증한다. 고위험 판단을 우선하는 보수적 판단 원칙을 적용한다.

### STEP 3. 통합·재검증·판단

- 검토 의견 통합
- 상충 판단 조정
- 고위험·저신뢰 이슈 재확인
- 위험 항목 생성·저장
- 심의자가 검토할 Evidence와 조치 방향 제시

## 8. Evidence 패널의 기대 형태

제안서의 예시 Evidence 패널은 다음 정보를 한 화면에서 보여준다.

- 위험도: `참고 / 주의 / 위험`
- 탐지 이슈: 예) 우대금리 조건 설명 부족
- 관련 기준: 예) 금융상품 광고 표현 기준
- 근거 자료: 예) 상품설명서 p.3 금리 조건
- 추천 조치: 예) 우대금리 적용 조건 문구 추가

UI에서는 업로드 자료와 규제·기준 근거를 구분해서 보여줘야 한다. 심의자가 근거의 출처, 버전, 적용일자, 관련 페이지를 빠르게 확인할 수 있어야 한다.

## 9. MVP 데모 시나리오

제안서의 대표 시나리오는 대출 홍보물 심의다. 별도의 핵심 기능 예시로 예·적금 홍보물 심의 화면도 제시되어 있으며, 두 예시 모두 `위험 지점 발견 -> 근거 연결 -> 수정 의견 초안 -> 심의자 최종 판단` 구조를 보여준다.

예·적금 홍보물 예시에서 확인되는 샘플 이슈:

- 예금자보호 문구가 상품설명서와 상충하는지 확인
- 최고금리 강조 문구에 조건부 적용 사실을 명확히 병기해야 하는지 확인
- 우대조건과 필수고지의 시각적 가독성이 매체별로 충분한지 확인
- `누구나`, `무조건`처럼 오인 가능성이 있는 단정 표현 확인
- 수정 의견 초안에는 대상 문구, 업로드 자료 근거, 수정 요청 문구가 함께 포함되어야 함

샘플 홍보물은 다음 유형의 표현을 포함한다.

- 누구나 빠르게 승인 가능한 최저금리 신용대출
- 최저 연 4.9%
- 앱에서 한도 확인 유도
- 베트남어 등 외국어 홍보 문구

이 시나리오에서 FinProof가 검토해야 하는 리스크는 다음과 같다.

- 최저금리 표시가 실제 적용 조건과 함께 충분히 설명되었는가
- 우대금리 조건과 상품설명서 내용이 일치하는가
- 승인 가능성을 과도하게 단정하거나 오인하게 만들지 않는가
- 대출 한도, 기간, 조건, 유의사항 등 필수 고지가 빠지지 않았는가
- 홍보물 문구와 약관·상품설명서·금리표가 충돌하지 않는가
- 외국어 문구의 뉘앙스가 한국어 기준으로 오인 가능성을 만들지 않는가
- `24시간 승인`, `무담보`, `승인률 98%` 같은 수치·단정 표현에 조건과 산정 근거가 충분히 붙어 있는가
- 반려 또는 수정 요청 의견이 실무자가 바로 전달할 수 있는 수준으로 정리되는가

시나리오 화면 흐름:

1. 요청자가 심의 요청 제목, 계열사, 요청 부서, 상품군, 게시 예정일, 게시 채널, 요청 메모를 입력한다.
2. 시스템이 업로드 파일을 자동 분류하고 누락 자료를 확인한다.
3. 심의자가 대기 목록에서 요청 건을 확인하고 AI 분석을 시작한다.
4. 워크벤치에서 AI가 찾은 이슈와 근거를 확인하고, RAG 채팅으로 추가 질의를 한다.
5. AI가 수정 요청 의견 초안을 생성한다.
6. 심의자가 최종 판단을 확정한다.
7. 요청자는 최종 심사 결과와 반려 사유 또는 수정 요청 의견을 확인한다.

## 10. 기술·데이터 구성

제안서와 기능명세서가 약속한 기술 구성은 다음과 같다.

| 계층        | 내용                                                                    | 주요 기술                                                 |
| ----------- | ----------------------------------------------------------------------- | --------------------------------------------------------- |
| 입력 데이터 | 홍보물, 약관, 상품설명서, 체크리스트, URL, 외부 자료, 금리표, 과거 사례 | PDF, DOCX, XLSX, URL, case data                           |
| AI 처리     | OCR, RAG 검색, 리랭킹, Rule Engine, LLM 판단·초안 생성                  | OCR provider, embedding, RAG, reranking, rule engine, LLM |
| 저장·검색   | 심의 근거 문서, 메타데이터, 원본 파일, 유사 사례, 감사 이력             | PostgreSQL, pgvector, Prisma, AWS S3                      |
| 운영·통제   | 인증, 권한, 접근 제어, 감사 이벤트, 배포 자동화                         | JWT/JWKS, RBAC, AuditEvent, EC2 + systemd, GitHub Actions |

운영 설계:

- Next.js 런타임과 분석 워커는 분리 배포한다.
- 분석 실행 모드는 `inline` 동기 실행과 `queued` 비동기 워커를 모두 지원한다.
- OCR, 임베딩, 리랭킹, LLM은 교체 가능한 provider 구조여야 한다.
- deterministic 모드는 시연, 테스트, 감사 재현성을 위해 유지한다.

## 11. 금융권 적용을 위한 통제 원칙

FinProof는 AI를 무조건 신뢰하는 구조가 아니라 통제형 AI로 설계되어야 한다.

- Human-in-the-loop: AI는 의견 초안을 제공하고, 준법심의자가 최종 조치를 확정한다.
- RBAC: requester, reviewer, compliance_admin 역할별 권한을 분리한다.
- AuditEvent: 자료 제출, AI 분석, 근거 확인, 최종 판단 이력을 추적 가능하게 저장한다.
- Conservative Decision: 고위험 또는 저신뢰 이슈는 재확인과 사람 검토를 우선한다.
- Evidence Guardrail: RAG 답변은 승인된 근거 범위 안에서만 해야 한다.

## 12. 기대 효과와 지표

제안서와 기능명세서의 기대 효과는 세 축이다.

- 준법심의자: 반복 검토 감소, 위험지점에서 근거와 판단으로 빠르게 이동, 의견 작성 부담 감소
- 요청자: 재요청 감소, 심의 진행 상태 가시화, 수정 요청사항 확인, 배포 적시성 향상
- 조직: 근거 기반 심의 품질 표준화, 감사 가능한 이력 축적, 규정 변경 대응력 향상

PoC 목표:

- 심의 1건당 1차 검토 시간 30% 이상 절감
- 고위험 표현과 필수 고지 누락 최소화
- 고위험 recall 우선
- AI 판단 근거 추적성 95% 이상

## 13. 확장 로드맵

MVP는 대출·예적금 금리 광고 중심으로 시작한다.

확장 방향:

- 전북은행·광주은행 예·적금·대출 홍보물 심의
- JB우리캐피탈, JB자산운용 등 계열사 상품 콘텐츠 확장
- 카드, 보험, 투자, 캐피탈 상품군으로 심의 범위 확장
- 앱, 웹, SNS, 카드뉴스, 랜딩페이지, 제휴 이벤트 등 다채널 콘텐츠 심의
- 베트남어, 미얀마어, 크메르어 등 글로벌·다국어 콘텐츠 심의
- 계열사·조직별 기준과 테넌트별 지식베이스 운영
- 마케팅·전자결재 시스템 연동
- 규제 변경 자동 스케줄링
- 성과 지표 자동 측정 대시보드

## 14. 앞으로 개발할 때 지켜야 할 기준

FinProof 개발에서 중요한 것은 멋진 AI 답변보다 `심의 업무가 끝까지 닫히는 것`이다. 기능을 추가할 때는 다음 질문을 기준으로 판단한다.

- 이 기능이 요청, 분석, 근거 확인, 최종 판단, 이력 저장 중 어느 단계에 연결되는가
- AI 산출물이 심의자의 최종 판단을 대체하지 않고 보조하는가
- 판단 근거가 출처, 문서 버전, 적용일자, 페이지 단위로 추적 가능한가
- 요청자와 심의자의 권한 경계가 유지되는가
- AuditEvent가 남는가
- 고위험·저신뢰 결과에 대해 보수적 재검증 경로가 있는가
- 데모에서 대출 홍보물 시나리오를 자연스럽게 설명할 수 있는가
- 공모전 주제인 Compliance AI의 문제의식인 심의 병목, 품질 편차, 다국어·다채널 확장, 규제 변경 대응을 직접 겨냥하는가
